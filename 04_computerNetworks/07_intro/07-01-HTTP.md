# 07-01-HTTP



1. HTTP Fundamentals
   
   What is HTTP?
   
   - HyperText Transfer Protocol - Application layer protocol for distributed hypermedia
     systems
   - Stateless protocol - Each request is independent
   - Client-Server model - Request/Response cycle
   - Text-based protocol - Human-readable (until HTTP/2)
   - Default port: 80 (HTTP), 443 (HTTPS)
   
   HTTP Request Structure
   
   GET /api/users/123 HTTP/1.1
   Host: api.example.com
   User-Agent: Mozilla/5.0
   Accept: application/json
   Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
   Cookie: sessionid=abc123
   Content-Type: application/json
   Content-Length: 45
   
   {"name": "John", "email": "john@example.com"}
   
   Components:
   
   1. Request Line: Method + URI + HTTP Version
   2. Headers: Key-value metadata
   3. Empty Line: Separator
   4. Body: Optional data payload
   
   HTTP Response Structure
   
   HTTP/1.1 200 OK
   Date: Mon, 02 Mar 2026 10:30:00 GMT
   Server: nginx/1.18.0
   Content-Type: application/json
   Content-Length: 87
   Cache-Control: max-age=3600
   ETag: "33a64df551425fcc55e4d42a148795d9f25f89d4"
   Set-Cookie: sessionid=xyz789; HttpOnly; Secure
   
   {"id": 123, "name": "John", "email": "john@example.com", "created_at": "2026-03-01"}
   
   ---
   
   2. HTTP Methods (Verbs)
   
   Core Methods
   
   | Method  | Purpose             | Idempotent | Safe | Cacheable        | Has Body |
   | ------- | ------------------- | ---------- | ---- | ---------------- | -------- |
   | GET     | Retrieve resource   | Yes        | Yes  | Yes              | No*      |
   | POST    | Create resource     | No         | No   | Only if explicit | Yes      |
   | PUT     | Replace resource    | Yes        | No   | No               | Yes      |
   | PATCH   | Partial update      | No         | No   | No               | Yes      |
   | DELETE  | Remove resource     | Yes        | No   | No               | No*      |
   | HEAD    | Get headers only    | Yes        | Yes  | Yes              | No       |
   | OPTIONS | Get allowed methods | Yes        | Yes  | No               | No       |
   | CONNECT | Establish tunnel    | No         | No   | No               | No       |
   | TRACE   | Echo request        | Yes        | Yes  | No               | No       |
   
   Key Concepts:
   
   - Idempotent: Multiple identical requests have same effect as single request
   - Safe: Read-only, doesn't modify server state
   - Cacheable: Response can be cached
   
   Practical Examples
   
   # GET - Retrieve resource
   
   curl -X GET https://api.example.com/users/123
   
   # POST - Create new resource
   
   curl -X POST https://api.example.com/users \
    -H "Content-Type: application/json" \
    -d '{"name":"John","email":"john@example.com"}'
   
   # PUT - Replace entire resource
   
   curl -X PUT https://api.example.com/users/123 \
    -H "Content-Type: application/json" \
    -d '{"name":"John Doe","email":"john@example.com","age":30}'
   
   # PATCH - Partial update
   
   curl -X PATCH https://api.example.com/users/123 \
    -H "Content-Type: application/json" \
    -d '{"age":31}'
   
   # DELETE - Remove resource
   
   curl -X DELETE https://api.example.com/users/123
   
   # HEAD - Get only headers (no body)
   
   curl -I https://api.example.com/users/123
   
   # OPTIONS - Check allowed methods (CORS preflight)
   
   curl -X OPTIONS https://api.example.com/users \
    -H "Origin: http://localhost:3000" \
    -H "Access-Control-Request-Method: POST"

   3. HTTP Status Codes
   
   Complete Status Code Categories
   
   1xx - Informational
   
   - 100 Continue - Server received request headers, client should send body
   - 101 Switching Protocols - Server switching to protocol requested by client
     (WebSocket upgrade)
   - 102 Processing - Server received and processing request (WebDAV)
   
   2xx - Success
   
   - 200 OK - Standard success response
   - 201 Created - New resource created successfully (return Location header)
   - 202 Accepted - Request accepted for processing, but not completed
   - 203 Non-Authoritative Information - Successful but modified by proxy
   - 204 No Content - Success but no body to return (DELETE operations)
   - 205 Reset Content - Success, client should reset form
   - 206 Partial Content - Partial GET request (range requests for video/large files)
   
   3xx - Redirection
   
   - 301 Moved Permanently - Resource permanently moved (update bookmarks)
   - 302 Found - Temporary redirect (location might change)
   - 303 See Other - Use GET to retrieve resource at another URI
   - 304 Not Modified - Cached version is still valid (conditional requests)
   - 307 Temporary Redirect - Temporary redirect, keep same HTTP method
   - 308 Permanent Redirect - Permanent redirect, keep same HTTP method
   
   4xx - Client Errors
   
   - 400 Bad Request - Malformed request syntax
   - 401 Unauthorized - Authentication required or failed
   - 403 Forbidden - Server understood but refuses (permissions)
   - 404 Not Found - Resource doesn't exist
   - 405 Method Not Allowed - HTTP method not supported for resource
   - 406 Not Acceptable - Can't satisfy Accept headers
   - 408 Request Timeout - Client took too long to send request
   - 409 Conflict - Request conflicts with current state (concurrent updates)
   - 410 Gone - Resource permanently deleted
   - 411 Length Required - Content-Length header required
   - 412 Precondition Failed - One of precondition headers failed (If-Match,
     If-None-Match)
   - 413 Payload Too Large - Request body too large
   - 414 URI Too Long - URL too long
   - 415 Unsupported Media Type - Content-Type not supported
   - 422 Unprocessable Entity - Syntax correct but semantically invalid
   - 429 Too Many Requests - Rate limit exceeded
   
   5xx - Server Errors
   
   - 500 Internal Server Error - Generic server error
   - 501 Not Implemented - Server doesn't support functionality
   - 502 Bad Gateway - Invalid response from upstream server
   - 503 Service Unavailable - Server temporarily unavailable (maintenance/overload)
   - 504 Gateway Timeout - Upstream server timeout
   - 505 HTTP Version Not Supported - HTTP version not supported
   
   Best Practices for Status Codes
   
   # Testing different status codes
   
   curl -w "\nHTTP Status: %{http_code}\n" https://api.example.com/endpoint
   
   # Test redirects
   
   curl -L -w "Final URL: %{url_effective}\n" https://example.com
   
   # Test rate limiting
   
   for i in {1..100}; do
    curl -w "%{http_code}\n" https://api.example.com/endpoint
   done

   5. HTTPS & TLS/SSL
   
   What is HTTPS?
   
   - HTTP over TLS/SSL (Transport Layer Security)
   - Encrypts data in transit
   - Provides authentication (server identity)
   - Ensures data integrity
   - Default port: 443
   
   TLS Handshake Process
   
   Client                                Server
    |                                      |
    |-------- Client Hello --------------->|
    |  (TLS version, cipher suites,       |
    |   random bytes, SNI)                |
    |                                      |
    |<------- Server Hello -----------------|
    |  (Chosen cipher, random bytes)      |
    |<------- Certificate ------------------|
    |  (Server cert + chain)              |
    |<------- Server Hello Done ------------|
    |                                      |
    |-------- Client Key Exchange -------->|
    |  (Encrypted pre-master secret)      |
    |-------- Change Cipher Spec --------->|
    |-------- Finished ------------------->|
    |                                      |
    |<------- Change Cipher Spec -----------|
    |<------- Finished ---------------------|
    |                                      |
    |======== Encrypted Data =============>|
   
   Certificate Components
   
   # View certificate details
   
   openssl s_client -connect example.com:443 -showcerts 2>/dev/null | \
    openssl x509 -noout -text
   
   # Key components:
   
   # - Subject: Who the certificate is issued to
   
   # - Issuer: Who issued the certificate (CA)
   
   # - Validity: Not Before / Not After dates
   
   # - Public Key: Server's public key
   
   # - Signature: CA's signature
   
   # - SAN (Subject Alternative Names): Additional domains
   
   # - Key Usage: Purpose of certificate
   
   Practical TLS/SSL Tasks
   
   # 1. Check certificate expiry
   
   echo | openssl s_client -connect example.com:443 2>/dev/null | \
    openssl x509 -noout -dates
   
   # 2. Check certificate chain
   
   openssl s_client -connect example.com:443 -showcerts
   
   # 3. Test specific TLS version
   
   openssl s_client -connect example.com:443 -tls1_2
   openssl s_client -connect example.com:443 -tls1_3
   
   # 4. Check supported cipher suites
   
   nmap --script ssl-enum-ciphers -p 443 example.com
   
   # 5. Test SNI (Server Name Indication)
   
   openssl s_client -connect 1.2.3.4:443 -servername example.com
   
   # 6. Check OCSP stapling
   
   openssl s_client -connect example.com:443 -status
   
   # 7. Verify certificate against CA bundle
   
   openssl s_client -connect example.com:443 -CApath /etc/ssl/certs
   
   # 8. Extract certificate from server
   
   echo | openssl s_client -connect example.com:443 2>/dev/null | \
    openssl x509 -out cert.pem
   
   # 9. Check certificate fingerprint
   
   openssl x509 -in cert.pem -noout -fingerprint -sha256
   
   # 10. Test SSL/TLS vulnerabilities
   
   sslscan example.com
   testssl.sh example.com
   
   Common TLS Issues
   
   # Issue 1: Certificate expired
   
   # Solution: Renew certificate (Let's Encrypt example)
   
   certbot renew
   
   # Issue 2: Certificate name mismatch
   
   # Check certificate SAN
   
   openssl s_client -connect example.com:443 2>/dev/null | \
    openssl x509 -noout -text | grep -A1 "Subject Alternative Name"
   
   # Issue 3: Incomplete certificate chain
   
   # Test chain
   
   openssl s_client -connect example.com:443 -showcerts
   
   # Issue 4: Mixed content (HTTPS page loading HTTP resources)
   
   # Check in browser DevTools -> Security tab
   
   # Issue 5: TLS version too old
   
   # Test minimum TLS version
   
   openssl s_client -connect example.com:443 -tls1

   6. HTTP/1.0 vs HTTP/1.1 vs HTTP/2 vs HTTP/3
   
   HTTP/1.0 (1996)
   
   - One request per connection
   - No persistent connections
   - No chunked encoding
   - No Host header (required for virtual hosting)
   
   HTTP/1.1 (1997)
   
   - Persistent connections (Connection: keep-alive)
   - Pipelining (send multiple requests without waiting)
   - Chunked transfer encoding
   - Host header mandatory
   - Range requests (partial content)
   - Better caching mechanisms
   
   # HTTP/1.1 request with keep-alive
   
   curl -H "Connection: keep-alive" https://example.com
   
   HTTP/2 (2015)
   
   - Binary protocol (not text-based)
   - Multiplexing (multiple requests over single connection)
   - Server push (proactive resource sending)
   - Header compression (HPACK)
   - Stream prioritization
   - Better performance (reduced latency)
   
   # Test HTTP/2 support
   
   curl -I --http2 https://www.google.com
   
   # Check HTTP version
   
   curl -I https://example.com -w "HTTP Version: %{http_version}\n"
   
   # Force HTTP/2
   
   curl --http2-prior-knowledge https://example.com
   
   HTTP/3 (2022)
   
   - Based on QUIC (UDP instead of TCP)
   - Faster connection establishment (0-RTT)
   - Better performance on lossy networks
   - Built-in encryption (TLS 1.3)
   - Connection migration (switch networks without reconnection)
   
   # Test HTTP/3 support
   
   curl --http3 https://cloudflare.com -I
   
   # Check if server supports HTTP/3
   
   curl -I https://example.com | grep -i alt-svc
   
   # Look for: alt-svc: h3=":443"
   
   Performance Comparison
   
   # Create test file for comparing versions
   
   cat > curl-format.txt << 'EOF'
      time_namelookup:  %{time_namelookup}s\n
   
         time_connect:  %{time_connect}s\n
   
      time_appconnect:  %{time_appconnect}s\n
     time_pretransfer:  %{time_pretransfer}s\n
   
        time_redirect:  %{time_redirect}s\n
   
   time_starttransfer:  %{time_starttransfer}s\n
   
                      ----------\n
           time_total:  %{time_total}s\n
   
   EOF
   
   # Test HTTP/1.1
   
   curl -w "@curl-format.txt" -o /dev/null -s https://example.com
   
   # Test HTTP/2
   
   curl --http2 -w "@curl-format.txt" -o /dev/null -s https://example.com
   
   # Test HTTP/3
   
   curl --http3 -w "@curl-format.txt" -o /dev/null -s https://cloudflare.com

   7. HTTP Cookies & Sessions
   
   Cookie Basics
   
   # Server sends cookie
   
   Set-Cookie: sessionid=abc123; HttpOnly; Secure; SameSite=Strict; Max-Age=3600;
   Path=/; Domain=.example.com
   
   # Client sends cookie back
   
   Cookie: sessionid=abc123; theme=dark
   
   Cookie Attributes
   
   | Attribute | Purpose                                    | Example
   
              |
   
   |-----------|--------------------------------------------|---------------------------
   ------------|
   | HttpOnly  | Prevent JavaScript access (XSS protection) | HttpOnly
   
              |
   
   | Secure    | Only send over HTTPS                       | Secure
   
              |
   
   | SameSite  | CSRF protection                            | SameSite=Strict
   
              |
   
   | Max-Age   | Lifetime in seconds                        | Max-Age=3600
   
              |
   
   | Expires   | Absolute expiration                        | Expires=Wed, 21 Oct 2026 
   07:28:00 GMT |
   | Path      | URL path scope                             | Path=/admin
   
              |
   
   | Domain    | Domain scope                               | Domain=.example.com
   
              |
   
   SameSite Values
   
   - Strict: Never send cross-site (most secure)
   - Lax: Send on top-level navigation (default)
   - None: Always send (requires Secure)
   
   Practical Cookie Examples
   
   # 1. Save cookies to file
   
   curl -c cookies.txt https://example.com/login \
    -d "username=user&password=pass"
   
   # 2. Use saved cookies
   
   curl -b cookies.txt https://example.com/dashboard
   
   # 3. Send custom cookie
   
   curl -b "sessionid=abc123" https://example.com
   
   # 4. View cookie details
   
   curl -c - https://example.com/login -d "username=user&password=pass"
   
   # 5. Test cookie attributes
   
   curl -I https://example.com/login | grep -i set-cookie

   8. HTTP Caching - Deep Dive
   
   Cache-Control Directives
   
   Request Directives
   
   Cache-Control: no-cache          # Revalidate before using cache
   Cache-Control: no-store          # Never cache
   Cache-Control: max-age=0         # Fetch fresh version
   Cache-Control: max-stale=3600    # Accept stale up to 1 hour old
   Cache-Control: min-fresh=600     # Must be fresh for at least 10 min
   Cache-Control: only-if-cached    # Only use cached version
   
   Response Directives
   
   Cache-Control: public                           # Any cache can store
   Cache-Control: private                          # Only browser cache
   Cache-Control: no-cache                         # Must revalidate
   Cache-Control: no-store                         # Never cache
   Cache-Control: max-age=3600                     # Fresh for 1 hour
   Cache-Control: s-maxage=7200                    # CDN cache for 2 hours
   Cache-Control: must-revalidate                  # Revalidate when stale
   Cache-Control: proxy-revalidate                 # Proxy must revalidate
   Cache-Control: immutable                        # Never changes (hash-based 
   filenames)
   Cache-Control: stale-while-revalidate=60        # Serve stale while fetching fresh
   Cache-Control: stale-if-error=86400             # Serve stale if server error
   
   Conditional Requests
   
   # Using ETag
   
   # 1st request
   
   curl -I https://api.example.com/data
   
   # Response: ETag: "abc123"
   
   # 2nd request with If-None-Match
   
   curl -H "If-None-Match: \"abc123\"" https://api.example.com/data
   
   # Response: 304 Not Modified (if unchanged)
   
   # Using Last-Modified
   
   # 1st request
   
   curl -I https://api.example.com/data
   
   # Response: Last-Modified: Wed, 21 Oct 2025 07:28:00 GMT
   
   # 2nd request with If-Modified-Since
   
   curl -H "If-Modified-Since: Wed, 21 Oct 2025 07:28:00 GMT"
   https://api.example.com/data
   
   # Response: 304 Not Modified (if unchanged)
   
   Cache Strategies
   
   1. Versioned URLs (Best for static assets)
      Cache-Control: public, max-age=31536000, immutable
      
      # Example: /static/app.abc123.js
   
   2. Short cache with revalidation (API responses)
      Cache-Control: private, max-age=300, must-revalidate
      ETag: "version-hash"
   
   3. No caching (Sensitive data)
      Cache-Control: no-store, no-cache, must-revalidate
      Pragma: no-cache
      Expires: 0
   
   4. Stale-while-revalidate (Better UX)
      Cache-Control: max-age=3600, stale-while-revalidate=86400
   
   Testing Cache Behavior
   
   # 1. Check cache headers
   
   curl -I https://example.com | grep -i cache
   
   # 2. Test if resource is cached
   
   curl -I https://example.com/style.css | grep -E "Cache-Control|ETag|Last-Modified"
   
   # 3. Force fresh fetch
   
   curl -H "Cache-Control: no-cache" https://example.com

   curl -H "If-None-Match: $ETAG" https://example.com/api -w "HTTP: %{http_code}\n"
   
   # 5. Check Vary header
   
   curl -I https://example.com | grep -i vary

   9. CORS (Cross-Origin Resource Sharing)
   
   CORS Preflight Request
   
   # Browser sends OPTIONS request
   
   OPTIONS /api/data HTTP/1.1
   Host: api.example.com
   Origin: http://localhost:3000
   Access-Control-Request-Method: POST
   Access-Control-Request-Headers: Content-Type, Authorization
   
   # Server responds
   
   HTTP/1.1 200 OK
   Access-Control-Allow-Origin: http://localhost:3000
   Access-Control-Allow-Methods: GET, POST, PUT, DELETE
   Access-Control-Allow-Headers: Content-Type, Authorization
   Access-Control-Max-Age: 86400
   Access-Control-Allow-Credentials: true
   
   CORS Scenarios
   
   Simple Request (No preflight needed):
   
   - Methods: GET, HEAD, POST
   - Headers: Accept, Accept-Language, Content-Language, Content-Type
   - Content-Type: application/x-www-form-urlencoded, multipart/form-data, text/plain
   
   Preflighted Request (Preflight needed):
   
   - Other methods: PUT, DELETE, PATCH
   - Custom headers: Authorization, X-Custom-Header
   - Content-Type: application/json
   
   Testing CORS
   
   # 1. Test simple CORS request
   
   curl -H "Origin: http://localhost:3000" \
    https://api.example.com/data -v
   
   # 2. Test preflight request
   
   curl -X OPTIONS https://api.example.com/data \
    -H "Origin: http://localhost:3000" \
    -H "Access-Control-Request-Method: POST" \
    -H "Access-Control-Request-Headers: Content-Type, Authorization" \
    -v
   
   # 3. Test actual POST request
   
   curl -X POST https://api.example.com/data \
    -H "Origin: http://localhost:3000" \
    -H "Content-Type: application/json" \
    -d '{"key":"value"}' \
    -v
   
   # 4. Check CORS headers in response
   
   curl -I -H "Origin: http://localhost:3000" https://api.example.com | grep -i
   access-control
   
   Common CORS Issues
   
   # Issue 1: Missing Access-Control-Allow-Origin
   
   # Error: No 'Access-Control-Allow-Origin' header is present
   
   # Solution: Server must include header
   
   # Issue 2: Credentials with wildcard origin
   
   # Error: Cannot use wildcard in Access-Control-Allow-Origin with credentials
   
   # Solution: Specify exact origin when using credentials
   
   # Issue 3: Missing allowed headers
   
   # Error: Request header field authorization is not allowed
   
   # Solution: Add to Access-Control-Allow-Headers
   
   # Issue 4: Method not allowed
   
   # Error: Method PUT is not allowed by Access-Control-Allow-Methods
   
   # Solution: Add method to Access-Control-Allow-Methods

   12. Advanced Topics for Senior Developers
   
   HTTP Pipelining
   
   # Send multiple requests without waiting for responses
   
   # (HTTP/1.1 feature, deprecated in HTTP/2)
   
   (echo -e "GET /page1 HTTP/1.1\r\nHost: example.com\r\n\r\n"; \
   echo -e "GET /page2 HTTP/1.1\r\nHost: example.com\r\n\r\n") | \
    nc example.com 80

   0\r\n
   \r\n

   # 1. Basic Authentication

   Common Issues & Solutions
   
   1. Slow Response Times
   
   # Diagnose slow requests
   
   curl -w "\nDNS Lookup: %{time_namelookup}s\nTCP Connect: %{time_connect}s\nTLS 
   Handshake: %{time_appconnect}s\nServer Processing: %{time_starttransfer}s\nTotal 
   Time: %{time_total}s\n" -o /dev/null -s https://example.com

   # - Use faster DNS servers (8.8.8.8, 1.1.1.1)
   
   # - Enable DNS caching
   
   # If TCP connect is slow:
   
   # - Network latency issue
   
   # - Check with: ping example.com
   
   # If TLS handshake is slow:
   
   # - Enable TLS session resumption
   
   # - Use OCSP stapling
   
   # If server processing is slow:
   
   # - Backend performance issue
   
   # - Check database queries
   
   # - Review application code
   
   2. Connection Refused
   
   # Check if port is open
   
   nc -zv example.com 443

   ss -tuln | grep :443

   3. SSL/TLS Errors
   
   # Certificate expired

   openssl s_client -connect example.com:443 2>&1 | openssl x509 -noout -text | grep -A
   1 "Subject Alternative Name"

   openssl s_client -connect example.com:443 -cipher 'HIGH:!aNULL:!MD5'
